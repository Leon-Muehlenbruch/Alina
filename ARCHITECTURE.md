# Architecture

## Overview

Alina is a serverless, privacy-first messenger built on the [Nostr](https://github.com/nostr-protocol/nostr) protocol. It runs entirely in the browser as a Progressive Web App. There is no backend, no database, no user accounts. All state lives on the user's device; messages are relayed through a federated network of public Nostr relays.

```
┌──────────────────────────────────────────────────────────────┐
│                        Browser (PWA)                         │
│                                                              │
│  React UI  →  Zustand Store  →  lib/nostr.ts  ←→  Relays   │
│                    ↕                                         │
│              localStorage                                    │
└──────────────────────────────────────────────────────────────┘
```

## Identity & Keys

Alina uses secp256k1 key pairs (the same curve as Bitcoin), provided by `nostr-tools`.

```
generateSecretKey()  →  privkey: Uint8Array (32 bytes)
getPublicKey(privkey)  →  pubkey: hex string (64 chars)
```

Keys are encoded using NIP-19 (bech32):
- `nsec1…` — private key (import/export)
- `npub1…` — public key (shareable identity)

The private key never leaves the device. It is stored in `localStorage` as a JSON array of bytes and loaded back into a `Uint8Array` at startup.

**Key storage layout** (`alina_identity` in localStorage):
```json
{
  "privkey": [12, 34, 56, ...],
  "pubkey": "a1b2c3...",
  "name": "Alice"
}
```

## Nostr Protocol Usage

Nostr is a simple publish/subscribe protocol over WebSocket. Clients connect to relays, publish signed JSON events, and subscribe to event streams by filter.

### Event Kinds Used

| Kind | Purpose | NIP |
|---|---|---|
| `4` | Direct message (E2E encrypted) | NIP-04 |
| `42` | Group room message (plaintext) | NIP-28 |
| `10420` | Invite code (custom, Alina-specific) | — |

### Relay Connections

Alina connects to 7 public relays simultaneously on startup:

```
wss://relay.damus.io
wss://nos.lol
wss://nostr.mom
wss://nostr.wine
wss://purplepag.es
wss://relay.primal.net
wss://relay.nostr.bg
```

Each relay gets its own WebSocket connection (`connectRelay` in `lib/nostr.ts`). On disconnect, automatic reconnection is attempted after 15 seconds. The live relay count is tracked in the Zustand store and shown as a colored dot in the sidebar (red / yellow / green).

### Subscriptions

On each relay connection, two subscriptions are opened:

```
REQ dm-sub    { kinds: [4],  "#p": [myPubkey],    limit: 100 }
REQ room-sub  { kinds: [42], "#e": [roomHash...], limit: 200 }
```

- `#p` filters DMs addressed to the user's public key
- `#e` filters room messages by room hash tag

### Message Flow (Incoming)

```
Relay WebSocket
  → handleRelayMessage()
    → if kind 4  → handleDM()  → decryptDM()  → onMessageCallback()
    → if kind 42 → handleRoomMsg()            → onMessageCallback()
    → if kind 10420 → pendingInviteLookups resolve
```

`onMessageCallback` is wired in `useNostrRelays.ts` and calls `store.addMessage()`, which deduplicates by `(ts, pubkey)` and persists to localStorage.

## Direct Messages (DMs)

DMs use **NIP-04**: ECDH shared secret derivation + AES-256-CBC encryption.

### Send Flow

```
ChatInput
  → publishDM(privkey, myPubkey, recipientPubkey, { type, content })
    → encryptDM()           // nip04.encrypt — ECDH + AES-256-CBC
    → createSignedEvent()   // kind 4, tags: [["p", recipientPubkey]]
    → publishToRelays()     // broadcast to all connected relays
    → store.addMessage()    // store own message locally
```

### Receive Flow

```
handleDM(event)
  → decryptDM(privkey, senderPubkey, content)  // nip04.decrypt
  → JSON.parse(decrypted)   // { type, content }
  → onMessageCallback("dm:" + senderPubkey, msg)
```

Message content is a JSON envelope:
```json
{ "type": "text", "content": "Hello!" }
{ "type": "image", "content": "data:image/png;base64,..." }
{ "type": "location", "content": "filled.count.soap" }
```

## Group Rooms

Group rooms are **not encrypted**. Anyone who knows the room name can join and read messages.

### Room Identity

The room name is hashed with SHA-256 to produce a deterministic room identifier:

```
SHA-256("alina-room-v1:" + name.toLowerCase())  →  roomHash (hex, 64 chars)
```

All members subscribe to `kind: 42` events tagged with `["e", roomHash]`. No server-side room creation is needed — the hash is the room.

### Send Flow

```
publishRoomMessage(privkey, myPubkey, roomHash, { type, content, name })
  → createSignedEvent()   // kind 42, tags: [["e", roomHash, "", "root"]]
  → publishToRelays()
  → store.addMessage()
```

The sender's display name is embedded in the message content (`name` field), so recipients see who sent each message even without a contact entry.

## Invite Codes

DM contacts are added via 6-digit one-time invite codes. This avoids the need to manually exchange public keys.

### Protocol

1. **Inviter** generates a 6-digit code and publishes a `kind 10420` event:
   ```
   tags: [["t", SHA-256("alina-invite-v1:" + code)], ["expiration", unixTs + 600]]
   content: JSON.stringify({ name: "Alice" })
   ```
   The code itself is never published — only its hash.

2. **Recipient** enters the code, looks it up:
   ```
   REQ invite-<random>  {
     kinds: [10420],
     "#t": [SHA-256("alina-invite-v1:" + code)],
     since: now - 600,
     limit: 5
   }
   ```
   On match: the inviter's `pubkey` and `name` are extracted and a new contact is created.

3. Lookup resolves in ≤8 seconds or returns `null` (timeout). Codes expire after 600 seconds.

## State Management

All application state lives in a single Zustand store (`src/store/useStore.ts`). State is divided into logical domains:

| Domain | Key state | Persistence |
|---|---|---|
| Identity | `identity` (privkey, pubkey, name) | localStorage |
| Contacts | `contacts: Record<pubkey, Contact>` | localStorage |
| Rooms | `rooms: Record<hash, Room>` | localStorage |
| Messages | `messages: Record<chatId, Message[]>` | localStorage |
| Unread | `unread: Record<chatId, number>` | localStorage |
| Relay | `relayCount` | in-memory only |
| Language | `lang: 'de' \| 'en' \| 'ru'` | localStorage (separate key) |
| Translation | `autoTranslate: boolean` | localStorage (separate key) |
| UI | `openModal`, `sidebarOpen`, `statusMessage` | in-memory only |

### Chat ID Convention

```
"dm:"   + pubkey    →  direct message thread
"room:" + roomHash  →  group room thread
```

### Message Deduplication

`addMessage` checks `(ts, pubkey)` before inserting. This handles the case where the same event arrives from multiple relays.

### Message Limit

Each chat keeps at most 200 messages (`MAX_MESSAGES_PER_CHAT`). Older messages are trimmed from the front.

## Internationalisation (i18n)

The app supports German (`de`), English (`en`), and Russian (`ru`).

### Architecture

- `src/lib/i18n.ts` — flat key-value dictionaries for all three languages. Keys are typed (`TKey`), so missing translations are caught at compile time.
- `src/hooks/useT.ts` — React hook that returns a `t(key, params?)` function. Subscribes to `lang` from the store, so all components re-render on language change.
- Parameter interpolation uses `{placeholder}` syntax: `t('sidebar.relays', { n: '3', total: '7' })`

Language preference is stored in `localStorage` under `alina-lang` and defaults to `'de'`.

## Auto-Translation

Incoming messages in foreign languages can be translated client-side. Encrypted content is decrypted first, and **only the decrypted plaintext is passed to the translation layer** — relay traffic is never touched.

### Provider Chain

```
translate(text, targetLang)
  1. Check localStorage cache (djb2 hash of text+lang → key)
  2. detectLang(text)
     a. Chrome AI Detector API (Chrome 127+, offline)
     b. Script heuristics fallback (Cyrillic→ru, CJK→zh, Arabic→ar, …)
  3. If detectedLang === targetLang → skip
  4. translateChromeAI(text, from, to)   // Chrome 127+, fully offline
     OR
     translateMyMemory(text, from, to)   // free REST API, 8s timeout
  5. Cache result in localStorage
```

### Privacy

- Chrome AI Translation: runs entirely offline. No data leaves the device.
- MyMemory: decrypted message text is sent to `api.mymemory.translated.net`. Used only as fallback when Chrome AI is unavailable.

## PWA & Offline Support

### Service Worker

Configured via `vite-plugin-pwa` with Workbox. Caching strategies:

- **App shell** (JS/CSS/HTML): CacheFirst, versioned
- **Google Fonts CSS**: StaleWhileRevalidate
- **Google Fonts files**: CacheFirst, 1 year TTL, max 30 entries

The service worker is registered in `src/main.tsx` via `registerSW()`.

### Install Prompt

`@khmyznikov/pwa-install` provides platform-specific install dialogs:

- **iOS Safari**: Step-by-step guide (Share → Add to Home Screen), since iOS never fires the native `beforeinstallprompt` event
- **Android Chrome / Desktop**: Native install dialog wrapper

The install button in the sidebar is only shown when the library fires `pwa-install-available-event`.

1. **SPA rewrites**: All routes except `/assets/` → `index.html`
2. **Security headers** on all routes:
   - `X-Content-Type-Options: nosniff`
   - `Referrer-Policy: strict-origin-when-cross-origin`
   - `Permissions-Policy: geolocation=(self)` (used for what3words location messages)

The app is fully static — no serverless functions, no API routes.

## Security Considerations

| Concern | Mitigation |
|---|---|
| Private key exposure | Stored only in localStorage; never transmitted; displayed as `nsec` only in Settings |
| DM confidentiality | NIP-04: ECDH shared secret + AES-256-CBC; relays see only ciphertext |
| Group room privacy | No encryption — treat as semi-public; documented in UI |
| Invite code brute-force | 10^6 possible codes; SHA-256 hashed on relays; 600s expiry |
| XSS via message content | React renders all content as text, never `dangerouslySetInnerHTML` |
| Image size | Base64 images capped at 500 KB client-side before sending |
| Build dependency CVE | `serialize-javascript` pinned to `≥7.0.5` via npm `overrides` |
